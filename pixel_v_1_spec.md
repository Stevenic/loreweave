# Pixel Format v1 Specification

The first pixel art format designed for LLM generation.

---

## 1. Overview

Pixel Format v1 is a deterministic, human-readable, JSON-based asset and scene format for retro pixel-art rendering in JavaScript and TypeScript game engines.

It is designed for:

- **LLM generation** — text IS the art; an LLM can read, write, and transform sprites as plain text
- **Reusable sprite assets** with optional internal layers and animation frames
- **Strict palette-based rendering** with semantic color vocabularies
- **Layered scene composition** with depth, parallax, blend modes, and tilemaps
- **Keyframe animation** for both frame sequences and property interpolation
- **Deterministic visual output** — the same file always renders the same image
- **Git diffing and hand editing** — every change is a meaningful text diff
- **Constrained generation** — hints that guide LLMs toward valid, high-quality output

The format is intended for multiplayer persistent worlds where the visual layer should be stable, composable, and renderer-controlled rather than regenerated by an image model each time.

### 1.1 Why text?

Binary pixel art formats (Aseprite `.ase`, Pyxel Edit `.pyxel`) are optimized for editor serialization. They are opaque to LLMs — impossible to generate, diff, or patch incrementally. Pixel Format v1 makes the opposite tradeoff: **the raw text is the art**. A single-character grid where each character maps to a palette color means an LLM literally "sees" the sprite shape in the token stream. This is the core design insight that everything else builds on.

### 1.2 Design principles

1. **Deterministic** — the same file always renders the same image.
2. **Composable** — sprites, tiles, and scenes combine cleanly.
3. **Palette-locked** — assets render from a known, constrained palette.
4. **Readable** — easy to inspect and edit as plain text.
5. **LLM-native** — optimized for generation, transformation, and reasoning by language models.
6. **Extensible** — optional fields never break existing parsers.
7. **JavaScript-friendly** — parses natively as JSON.

## 2. File Types

Pixel Format v1 defines six JSON-based file types.

| Type | Format String | Extension | Purpose |
|------|--------------|-----------|---------|
| Sprite | `pixel-sprite-v1` | `.pixel.json` | Individual reusable asset with optional layers and animation |
| Tileset | `pixel-tileset-v1` | `.tileset.pixel.json` | Named collection of tile-sized assets |
| Tilemap | `pixel-tilemap-v1` | `.tilemap.pixel.json` | Grid-based tile placement with transforms |
| Scene | `pixel-scene-v1` | `.scene.pixel.json` | Composition of assets onto a canvas with layers, camera, and effects |
| Palette | `pixel-palette-v1` | `.palette.pixel.json` | Shared color definitions with aliases and ramps |
| Emitter | `pixel-emitter-v1` | `.emitter.pixel.json` | Particle emitter configuration |

Every file must declare a `format` string. Renderers must reject unknown format values unless explicitly configured for forward compatibility.

---

## 3. Coordinate System

All coordinates use a **top-left origin**.

- `x` increases left to right
- `y` increases top to bottom

Sprites may define an `origin` anchor point for placement in scenes (e.g., bottom-center for characters, center for projectiles).

## 4. Palette Specification

A palette maps single-character symbol keys to colors. Palettes are the vocabulary that constrains LLM output — instead of guessing hex values, an LLM picks from a curated set of named characters.

### 4.1 Color values

Palette entries must be one of:

- `"transparent"` → RGBA `(0, 0, 0, 0)`
- `"#RRGGBB"` → alpha defaults to `FF`
- `"#RRGGBBAA"` → explicit alpha

### 4.2 Allowed symbol keys

Palette keys are single characters:

- Digits: `0-9`
- Uppercase letters: `A-Z`
- Lowercase letters: `a-z`
- Symbols: `! @ # $ % & * + = ?`

Maximum 72 palette entries. Every character used in pixel data must exist in the active palette.

### 4.3 Semantic aliases

Palettes may include an `aliases` object that maps palette keys to human-readable names:

```json
"aliases": {
  "1": "bark-dark",
  "2": "bark-mid",
  "3": "bark-light",
  "6": "leaf-dark",
  "7": "leaf-mid",
  "8": "leaf-light"
}
```

Aliases serve two purposes:
1. **LLM guidance** — an LLM generating a tree sprite can reason about "bark-dark" vs "leaf-light" instead of memorizing that `1` means brown
2. **Documentation** — human editors see intent, not just hex codes

Aliases are optional metadata. They do not affect rendering. Keys in `aliases` must exist in `entries`.

### 4.4 Color ramps

Palettes may include a `ramps` object that defines ordered color gradients:

```json
"ramps": {
  "bark": ["1", "2", "3"],
  "leaf": ["6", "7", "8", "9"],
  "skin": ["D", "E"]
}
```

Each ramp is an array of palette keys ordered from darkest to lightest. Ramps give LLMs a vocabulary for shading — "use the next key in the bark ramp for highlights" is a natural instruction.

Ramps are optional metadata. They do not affect rendering. All keys in ramps must exist in `entries`.

### 4.5 Palette file example

  "name": "fantasy16",
  "entries": {

    "A": "#6d7a8a",

    "C": "#d8e2f1",
    "D": "#7f2f35",
    "E": "#bf5a5a",
    "F": "#10151f"
  },
  "aliases": {
    "0": "transparent",
    "1": "bark-dark",
    "2": "bark-mid",
    "3": "bark-light",

### 4.6 Palette resolution

A sprite or tileset `palette` field may be:

1. **A string name** — resolved from a runtime palette registry
2. **An inline object** — `entries` (and optionally `aliases`, `ramps`) embedded directly

If a string name is used and resolution fails, rendering must fail with an explicit error.

## 5. Sprite Specification

A sprite is the atomic unit of the format. Everything else references sprites. A sprite contains palette-indexed pixel data as a character grid — the text IS the art.

### 5.1 Required fields

| Field | Type | Description |
|-------|------|-------------|
| `format` | `"pixel-sprite-v1"` | Format identifier |
| `name` | `string` | Unique asset name |
| `width` | `integer > 0` | Pixel width of one frame |
| `height` | `integer > 0` | Pixel height of one frame |
| `palette` | `string \| object` | Palette name or inline palette |
| `encoding` | `"raw" \| "rle"` | Pixel data encoding |
| `pixels` | `string[]` | Pixel data rows (flat sprite or all frames vertically stacked) |

### 5.2 Optional fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `layers` | `SpriteLayer[]` | — | Named pixel layers (replaces `pixels` when present) |
| `frameCount` | `integer > 0` | `1` | Number of animation frames |
| `clips` | `Record<string, Clip>` | — | Named animation clips |
| `origin` | `{x, y}` | `{x: 0, y: 0}` | Anchor point for scene placement |
| `ppu` | `integer > 0` | `32` | Pixels Per Unit — how many pixels represent one world tile |
| `baseScale` | `number > 0` | `1.0` | Intended rendering scale for depth composition |
| `symmetry` | `"none" \| "horizontal" \| "vertical" \| "both"` | `"none"` | Symmetry hint for generators |
| `regions` | `Record<string, Region>` | — | Named rectangular regions |
| `attachments` | `Record<string, Attachment>` | — | Named attachment points for composition |
| `constraints` | `Constraints` | — | Generation quality hints |
| `variants` | `Record<string, Variant>` | — | Declarative variant rules |
| `generation` | `GenerationMeta` | — | Provenance metadata |
| `tags` | `string[]` | — | Classification tags |
| `collision` | `{x, y, width, height}` | — | Collision box |
| `meta` | `object` | — | Arbitrary metadata |

### 5.3 Flat sprite example

The simplest form — a single-frame sprite with inline palette:

```json
{
  "format": "pixel-sprite-v1",
  "name": "campfire_small",
  "width": 8,
  "height": 8,
  "palette": {
    "entries": {
      "0": "transparent",
      "1": "#2b1d0e",
      "3": "#8b5a2b",
      "4": "#e2a13b",
      "5": "#ffd46b"
    }
  },
  "encoding": "raw",
  "pixels": [
    "00044000",
    "00045500",
    "00044500",
    "00035300",
    "00313130",
    "00111110",
    "000AA000",
    "00000000"
  ]
}
```

An LLM can read this and see the flame shape directly in the text. That's the point.

### 5.4 Sprite layers

When `layers` is present, it replaces `pixels`. Each layer has its own pixel data and is composited in array order (first = back, last = front):

```json
{
  "format": "pixel-sprite-v1",
  "name": "potion_red",
  "width": 8,
  "height": 8,
  "palette": {
    "entries": {
      "0": "transparent",
      "1": "#1a1a2e",
      "2": "#e94560",
      "3": "#ff6b6b",
      "4": "#ffffff"
    }
  },
  "encoding": "raw",
  "layers": [
    {
      "name": "outline",
      "pixels": [
        "00011000",
        "00100100",
        "01000010",
        "01000010",
        "10000001",
        "10000001",
        "10000001",
        "01111110"
      ]
    },
    {
      "name": "fill",
      "pixels": [
        "00000000",
        "00044000",
        "00334300",
        "00233200",
        "02222220",
        "02222220",
        "02222220",
        "00000000"
      ]
    }
  ]
}
```

Layer rules:
- Each layer must have `name` (string) and `pixels` (string array)
- Each layer's pixel data must match the sprite's `width` and `height` (× `frameCount` if animated)
- Layers composite in array order: first layer is the back, last is the front
- Transparent pixels (`0` mapped to `"transparent"`) do not overwrite pixels from layers below
- Optional per-layer `opacity` (0.0–1.0, default 1.0)

If both `pixels` and `layers` are present, `layers` takes precedence and `pixels` is ignored.

### 5.5 Animation frames

Sprites can contain multiple animation frames via vertical stacking. When `frameCount` is set:

- Total pixel rows = `height × frameCount`
- Frames are sliced top-to-bottom: rows 0–(height-1) = frame 0, rows height–(2×height-1) = frame 1, etc.
- Each frame uses the same `width`

```json
{
  "format": "pixel-sprite-v1",
  "name": "torch_flame",
  "width": 4,
  "height": 6,
  "frameCount": 3,
  "palette": "fantasy16",
  "encoding": "raw",
  "pixels": [
    "0450",
    "0540",
    "4554",
    "0440",
    "0110",
    "0110",

    "0540",
    "4550",
    "0454",
    "0440",
    "0110",
    "0110",

    "0050",
    "0450",
    "4544",
    "0440",
    "0110",
    "0110"
  ]
}
```

If `layers` is used with `frameCount`, each layer's pixel data contains `height × frameCount` rows.

### 5.6 Animation clips

Clips define named animation sequences. Each clip contains keyframe tracks for frame selection and/or property animation:

```json
"clips": {
  "flicker": {
    "duration": 450,
    "playback": "loop",
    "tracks": [
      {
        "property": "frame",
        "keyframes": [
          { "time": 0, "value": 0 },
          { "time": 150, "value": 1 },
          { "time": 300, "value": 2 }
        ]
      }
    ]
  },
  "extinguish": {
    "duration": 500,
    "playback": "once",
    "tracks": [
      {
        "property": "frame",
        "keyframes": [
          { "time": 0, "value": 2 },
          { "time": 250, "value": 1 },
          { "time": 400, "value": 0 }
        ]
      },
      {
        "property": "opacity",
        "keyframes": [
          { "time": 0, "value": 1.0 },
          { "time": 500, "value": 0.0 }
        ],
        "easing": "ease-out"
      }
    ]
  }
}
```

#### Clip fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `duration` | `number` | required | Total clip length in milliseconds |
| `playback` | `"loop" \| "once" \| "pingpong"` | `"loop"` | Playback mode |
| `tracks` | `Track[]` | required | Array of property tracks |

#### Track fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `property` | `string` | required | Property to animate (see below) |
| `keyframes` | `Keyframe[]` | required | Ordered time-value pairs |
| `easing` | `string` | `"step"` for `frame`, `"linear"` for others | Interpolation function |

#### Keyframe fields

| Field | Type | Description |
|-------|------|-------------|
| `time` | `number` | Time in ms from clip start (0 ≤ time ≤ duration) |
| `value` | `number \| string` | Property value at this keyframe |

#### Animatable properties

| Property | Type | Interpolation | Description |
|----------|------|--------------|-------------|
| `frame` | `integer` | step | Which frame index to display |
| `offsetX` | `number` | continuous | Horizontal offset from placement point |
| `offsetY` | `number` | continuous | Vertical offset from placement point |
| `rotation` | `number` | continuous | Rotation in degrees (clockwise) |
| `scale` | `number` | continuous | Uniform scale multiplier |
| `opacity` | `number` | continuous | Transparency (0.0–1.0) |

#### Easing functions

| Easing | Description |
|--------|-------------|
| `step` | Jump to value instantly (default for `frame`) |
| `linear` | Constant rate interpolation (default for continuous properties) |
| `ease-in` | Accelerate from zero velocity |
| `ease-out` | Decelerate to zero velocity |
| `ease-in-out` | Accelerate then decelerate |

The renderer computes intermediate values between keyframes using the specified easing function. For `frame` tracks, **step interpolation** means the renderer shows the frame specified at each keyframe until the next keyframe time — it does NOT blend between pixel grids (that would destroy the pixel art aesthetic). For continuous properties like `offsetX`, `rotation`, `scale`, and `opacity`, the renderer mathematically interpolates between keyframe values, producing smooth motion without requiring the LLM to specify every intermediate state.

### 5.7 Symmetry hints

The `symmetry` field tells generators they only need to produce half (or quarter) of the pixel data:

| Value | Generator behavior | Renderer behavior |
|-------|-------------------|-------------------|
| `"none"` | Generate full grid | Render as-is |
| `"horizontal"` | Generate left half; right half mirrors | Render as-is (full data stored) |
| `"vertical"` | Generate top half; bottom half mirrors | Render as-is (full data stored) |
| `"both"` | Generate top-left quarter; rest mirrors | Render as-is (full data stored) |

**Full pixel data is always stored in the file.** The symmetry hint is advisory — it tells generators and tooling that the sprite is intended to be symmetrical. A generator can produce only the unique portion and a post-processing tool mirrors it to fill the complete grid. Validators may warn (but not reject) if actual pixel data doesn't match the declared symmetry.

This optimization is significant for LLMs: a 16×16 sprite with `"horizontal"` symmetry requires generating only 128 characters instead of 256 — a 50% token reduction.

### 5.8 Named regions

Regions define named rectangular areas within a sprite:

```json
"regions": {
  "head": { "x": 2, "y": 0, "width": 4, "height": 4 },
  "body": { "x": 1, "y": 4, "width": 6, "height": 6 },
  "weapon_hand": { "x": 7, "y": 5, "width": 2, "height": 3 }
}
```

Regions enable:
- **Partial regeneration** — "regenerate just the head region"
- **Multiple hitboxes** — "head takes 2x damage, body takes 1x"
- **LLM-guided composition** — "place a helmet sprite at the head region"
- **Tooling** — editors can highlight and isolate regions

Region coordinates are relative to the sprite's top-left corner. Regions may overlap.

### 5.9 Attachment points

Attachments define named positions where other sprites can connect:

```json
"attachments": {
  "hand_right": { "x": 12, "y": 8 },
  "hand_left": { "x": 2, "y": 8 },
  "head_top": { "x": 7, "y": 0 },
  "feet": { "x": 7, "y": 15 }
}
```

Attachment points enable lightweight composition without a full skeleton system. A scene can place a sword sprite at a character's `hand_right` attachment. If the sprite has animation frames, attachment positions can vary per frame by adding an optional `frames` field:

```json
"attachments": {
  "hand_right": {
    "x": 12, "y": 8,
    "frames": {
      "1": { "x": 13, "y": 7 },
      "2": { "x": 14, "y": 6 }
    }
  }
}
```

The base `x, y` is the default. Frame-specific overrides apply when that frame is active. Unlisted frames use the base position.

### 5.10 Generation constraints

Constraints are advisory hints for LLM generators:

```json
"constraints": {
  "maxColors": 6,
  "outline": true,
  "noIslands": true,
  "connectedPixels": true
}
```

| Constraint | Type | Description |
|-----------|------|-------------|
| `maxColors` | `integer` | Maximum distinct palette keys (excluding transparent) |
| `outline` | `boolean` | Sprite should have a 1px dark outline around non-transparent regions |
| `noIslands` | `boolean` | No isolated single-pixel regions (reduces noise) |
| `connectedPixels` | `boolean` | All non-transparent pixels should form a single connected region |

Constraints are **advisory** — generators should respect them, validators may warn on violations, but files are never rejected for constraint violations. They exist to help LLMs produce cleaner pixel art.

### 5.11 Variant rules

Variants declare derived versions of a sprite without duplicating pixel data:

```json
"variants": {
  "potion_blue": {
    "palette": {
      "2": "#4560e9",
      "3": "#6b8bff"
    }
  },
  "potion_large": {
    "scale": 2.0
  },
  "potion_blue_large": {
    "palette": {
      "2": "#4560e9",
      "3": "#6b8bff"
    },
    "scale": 1.5
  }
}
```

| Variant field | Type | Description |
|--------------|------|-------------|
| `palette` | `Record<string, PixelColor>` | Palette key overrides (partial swap) |
| `scale` | `number` | Scale multiplier override |

Variants are resolved at load time. The renderer applies palette overrides before rendering, and scale overrides multiply with `baseScale`. This is the canonical mechanism for palette swaps (e.g., same potion in different colors).

### 5.12 Pixels Per Unit (PPU)

The `ppu` field declares how many pixels in the sprite represent one world tile (unit). This anchors pixel dimensions to world-space proportions — the "DPI for a tile-based world."

**Formula:** `world size (tiles) = pixel dimensions ÷ ppu`

```json
"ppu": 32
```

| Asset Class | Pixel Size | PPU | World Size (tiles) |
|-------------|-----------|-----|-------------------|
| Icon / tiny object | 16×16 | 64 | 0.25 × 0.25 |
| Potion bottle | 16×16 | 32 | 0.5 × 0.5 |
| Person | 32×48 | 32 | 1 × 1.5 |
| Lodge front | 64×64 | 16 | 4 × 4 |
| Large tree | 32×64 | 32 | 1 × 2 |
| Terrain tile | 32×32 | 8 | 4 × 4 |

#### PPU reference values

| PPU | Best For |
|-----|----------|
| 64 | Icons, tiny UI elements (sub-tile objects) |
| **32** | **Characters, items, props (default)** |
| 16 | Buildings, large structures |
| 8 | Massive terrain, world map tiles |

At 1× rendering, one world tile = `ppu` screen pixels. A person (32×48 at ppu 32) renders at 32×48 screen pixels — exactly 1 tile wide. A lodge (64×64 at ppu 16) renders at 64×64 screen pixels — but occupies 4×4 tiles in world space. The proportions are correct without manual scale math.

#### Scene-level auto-scaling with `referencePpu`

Scenes may declare a `referencePpu` on the canvas. When present, items whose sprites have a different `ppu` are automatically scaled so that world-space proportions are consistent:

```
auto-scale factor = sprite.ppu / canvas.referencePpu
```

For example, with `referencePpu: 32`:
- A character sprite (ppu 32) renders at 1× — no adjustment
- A lodge sprite (ppu 16) renders at 0.5× — its 64px width displays at 32px per tile, matching the character's scale

This removes the need for manual `item.scale` adjustments when composing assets with different detail levels.

#### LLM generation guidance

Generation prompts should include `ppu` so the LLM knows the pixel budget: *"Generate a lodge front view at ppu 16, 64×64 pixels (4×4 tiles)"*. The LLM knows exactly how many pixels to use, how detailed each one should be, and how this asset relates to other assets in the world.

### 5.13 Base scale

The `baseScale` field declares a sprite's intended rendering size relative to other sprites:

```json
"baseScale": 1.0
```

If a house is drawn at 32×32 pixels and a potion at 8×8, they can both declare `baseScale: 1.0` to indicate they're drawn at their intended world size. A sprite drawn at 2× detail for close-up use would set `baseScale: 0.5` to render at half size in normal scenes.

The effective rendering scale is: `canvas.scale × item.scale × sprite.baseScale`

### 5.14 Generation metadata

Provenance tracking for LLM-generated assets:

```json
"generation": {
  "prompt": "An 8x8 red healing potion bottle with a cork, fantasy style",
  "model": "claude-sonnet-4-5-20250514",
  "seed": 42,
  "parent": "potion_generic",
  "timestamp": "2026-03-26T19:00:00Z"
}
```

| Field | Type | Description |
|-------|------|-------------|
| `prompt` | `string` | The generation prompt |
| `model` | `string` | Model identifier |
| `seed` | `number` | RNG seed for reproducibility |
| `parent` | `string` | Parent sprite name (for variant lineage) |
| `timestamp` | `string` | ISO 8601 generation time |

All fields are optional. This metadata has no effect on rendering — it exists for asset management and lineage tracking.

## 6. Encoding

### 6.1 Raw encoding

When `encoding` is `"raw"`, `pixels` is an array of strings where each character is one palette key:

```json
"encoding": "raw",
"pixels": [
  "00044000",
  "00045500",
  "00044500",
  "00000000"
]
```

Rules:
- Array length must equal `height` (× `frameCount` if animated)
- Each row length must equal `width`
- Each character is one palette key
- Every key must exist in the active palette

### 6.2 RLE encoding

When `encoding` is `"rle"`, each row is a string of space-separated run tokens:

```
<count><symbol>
```

Examples:
- `8A` = eight pixels of key `A`
- `30` = three transparent pixels
- `12B` = twelve pixels of key `B`

Row example:
```
3A 2B 3A
```
Decodes to: `AAABBAAA`

Rules:
- Tokens separated by spaces
- Total decoded width per row must equal `width`
- Row count must equal `height` (× `frameCount` if animated)
- Counts must be positive integers
- Zero-length runs are invalid
- Adjacent same-symbol runs should be merged (`2A` not `1A 1A`)

### 6.3 RLE example

```json
{
  "format": "pixel-sprite-v1",
  "name": "fog_band",
  "width": 16,
  "height": 4,
  "palette": {
    "entries": {
      "0": "transparent",
      "B": "#d8e2f180"
    }
  },
  "encoding": "rle",
  "pixels": [
    "16B",
    "2B 12B 2B",
    "1B 14B 1B",
    "16B"
  ]
}
```

### 6.4 Decoding algorithm

```
1. Split each row by whitespace
2. Parse each token as /^(\d+)(.)$/
3. Expand: repeat the symbol by count
4. Concatenate runs into one decoded row string
5. Verify decoded row length equals width
```

### 6.5 Encoding recommendation

For consistent diffs, writers should:
- Merge adjacent runs of the same symbol
- Use a single space between tokens
- Avoid `1A 1A`; emit `2A` instead

## 7. Tileset Specification

Tilesets organize multiple tile-sized assets under a common palette.

### 7.1 Required fields

| Field | Type | Description |
|-------|------|-------------|
| `format` | `"pixel-tileset-v1"` | Format identifier |
| `name` | `string` | Unique tileset name |
| `tileWidth` | `integer > 0` | Width of each tile |
| `tileHeight` | `integer > 0` | Height of each tile |
| `palette` | `string \| object` | Shared palette |
| `tiles` | `Record<string, Tile>` | Named tile entries |

Optional tileset fields:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `ppu` | `integer > 0` | `32` | Pixels Per Unit for all tiles in this tileset |
| `meta` | `object` | — | Arbitrary metadata |

Each tile provides:
- `encoding` — `"raw"` or `"rle"`
- `pixels` — pixel data rows (dimensions = tileWidth × tileHeight)
- `tags` (optional) — classification tags
- `meta` (optional) — arbitrary metadata

### 7.2 Tile connectivity

For tiles that connect to neighbors (walls, paths, water edges), the optional `connectivity` field declares Wang tile edge matching:

```json
"tiles": {
  "path_straight": {
    "encoding": "raw",
    "pixels": ["..."],
    "connectivity": {
      "north": "path",
      "east": "path",
      "south": "path",
      "west": "path"
    }
  },

    "encoding": "raw",
    "pixels": ["..."],
    "connectivity": {
      "north": "path",
      "east": "path",
      "south": "grass",
      "west": "grass"
    }
  }
}
```

Edge labels are arbitrary strings. The tilemap placement engine (or an LLM) matches edges between adjacent tiles so that `tile_a.east == tile_b.west` at every boundary. This enables auto-tiling and constrained procedural placement.

## 8. Tilemap Specification

Tilemaps define grid-based tile placement — a 2D map built from tileset references.

### 8.1 Required fields

| Field | Type | Description |
|-------|------|-------------|
| `format` | `"pixel-tilemap-v1"` | Format identifier |
| `name` | `string` | Unique tilemap name |
| `tileset` | `string` | Tileset name reference |
| `gridWidth` | `integer > 0` | Grid columns |
| `gridHeight` | `integer > 0` | Grid rows |
| `legend` | `Record<string, string \| TileRef>` | Maps single characters to tile names |
| `grid` | `string[]` | Character grid — each char maps through legend to a tile |

### 8.2 Tile legend

The legend maps single characters to tile names (from the referenced tileset):

```json
"legend": {
  "G": "grass",
  "D": "dirt_path",
  "W": "water",
  "S": "stone"
}
```

### 8.3 Tile transforms

Legend entries can specify transforms to avoid duplicating tile variants:

```json
"legend": {
  "G": "grass",
  "D": "dirt_path",
  "d": { "tile": "dirt_path", "flipX": true },
  "R": { "tile": "dirt_path", "rotate": 90 }
}
```

| Transform | Type | Description |
|-----------|------|-------------|
| `tile` | `string` | Tile name from tileset |
| `flipX` | `boolean` | Mirror horizontally |
| `flipY` | `boolean` | Mirror vertically |
| `rotate` | `0 \| 90 \| 180 \| 270` | Clockwise rotation in degrees |

### 8.4 Tilemap example

```json
{
  "format": "pixel-tilemap-v1",
  "name": "forest_path",
  "tileset": "forest_ground",
  "gridWidth": 10,
  "gridHeight": 6,
  "legend": {
    "G": "grass",
    "D": "dirt_path",
    "W": "water",
    "S": "stone"
  },
  "grid": [
    "GGGGGGGGGG",
    "GGGDDGGGGG",
    "GGDDDDGGWW",
    "GDDDDDDGWW",
    "GGDDDDGGGG",
    "GGGDDGGGGG"
  ]
}
```

The character grid is the map layout — an LLM can "see" the path winding through grass, just like it sees sprite shapes in pixel data. Same design principle: **text IS the map**.

## 9. Scene Specification

Scenes compose sprites, tilesets, and tilemaps onto a canvas with layers, camera control, blend modes, parallax, and depth sorting.

### 9.1 Required fields

| Field | Type | Description |
|-------|------|-------------|
| `format` | `"pixel-scene-v1"` | Format identifier |
| `name` | `string` | Unique scene name |
| `canvas` | `Canvas` | Canvas dimensions and settings |
| `layers` | `SceneLayer[]` | Ordered array of layers |

### 9.2 Canvas

```json
"canvas": {
  "width": 256,
  "height": 192,
  "background": "#89b9ff",
  "scale": 1,
  "camera": {
    "x": 128,
    "y": 96,
    "zoom": 1.0
  }
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `width` | `integer > 0` | required | Canvas width in pixels |
| `height` | `integer > 0` | required | Canvas height in pixels |
| `background` | `PixelColor` | `"transparent"` | Background fill color |
| `scale` | `number > 0` | `1` | Global scale multiplier |
| `referencePpu` | `integer > 0` | — | Reference PPU for auto-scaling items (see §5.12) |
| `camera` | `Camera` | — | Viewport position and zoom |

#### Camera

The optional `camera` defines a viewport into the scene. When present:
- `x, y` = center of the viewport in scene coordinates
- `zoom` = additional zoom multiplier (1.0 = normal)
- The visible area = `(width / zoom) × (height / zoom)` pixels centered at `(x, y)`

This enables scenes larger than the visible area with pan and zoom control.

### 9.3 Layers

Layers render in array order (first = back, last = front). Each layer has a type:

#### Item layers (default)

```json
{
  "name": "objects",
  "items": [
    { "asset": "pine_tree", "x": 18, "y": 70 },
    { "asset": "campfire_small", "x": 82, "y": 76 }
  ]
}
```

#### Tilemap layers

```json
{
  "name": "terrain",
  "type": "tilemap",
  "tilemap": "forest_path"
}
```

A tilemap layer references a tilemap by name. The tilemap is rendered at position `(0, 0)` of the layer (offset by layer-level `offsetX`, `offsetY` if present).

#### Layer fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `name` | `string` | required | Layer name |
| `type` | `"items" \| "tilemap"` | `"items"` | Layer type |
| `items` | `SceneItem[]` | — | Items (for `"items"` type) |
| `tilemap` | `string` | — | Tilemap reference (for `"tilemap"` type) |
| `opacity` | `number` | `1.0` | Layer opacity (0.0–1.0) |
| `blend` | `BlendMode` | `"normal"` | Compositing blend mode |
| `scrollFactor` | `number` | `1.0` | Parallax scroll factor |
| `sort` | `"manual" \| "y"` | `"manual"` | Depth sorting mode |
| `offsetX` | `number` | `0` | Layer horizontal offset |
| `offsetY` | `number` | `0` | Layer vertical offset |
| `visible` | `boolean` | `true` | Layer visibility toggle |

#### Blend modes

| Mode | Description |
|------|-------------|
| `normal` | Standard alpha compositing (default) |
| `multiply` | Darkens — good for shadows, tinting |
| `screen` | Lightens — good for glow, fog |
| `overlay` | Combines multiply and screen — good for lighting |
| `additive` | Adds color values — good for fire, magic effects |

Blend modes match Canvas 2D `globalCompositeOperation` values for direct browser implementation.

#### Parallax

The `scrollFactor` controls how fast a layer scrolls relative to camera movement:

```json
{ "name": "distant_mountains", "scrollFactor": 0.2, "items": [...] },
{ "name": "midground_trees",   "scrollFactor": 0.6, "items": [...] },
{ "name": "foreground",        "scrollFactor": 1.0, "items": [...] }
```

When the camera pans, each layer moves `pan × scrollFactor` pixels. Distant layers (`< 1.0`) move slowly, creating depth. This is the classic 2D parallax technique.

#### Y-sort depth

When `sort` is `"y"`, items within the layer are sorted by `y` position before rendering (lower y = behind, higher y = in front). This is how classic RPGs (Zelda, Pokémon) handle character-vs-object depth. When `sort` is `"manual"` (default), items render in array order, with optional `z` for stable sorting.

### 9.4 Scene items

Each item in an items-layer places a sprite at a position:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `asset` | `string` | required | Sprite or tileset member name |
| `x` | `number` | required | X position on canvas |
| `y` | `number` | required | Y position on canvas |
| `scale` | `number` | `1.0` | Item scale multiplier |
| `opacity` | `number` | `1.0` | Item opacity (0.0–1.0) |
| `flipX` | `boolean` | `false` | Mirror horizontally |
| `flipY` | `boolean` | `false` | Mirror vertically |
| `rotation` | `number` | `0` | Clockwise rotation in degrees |
| `z` | `number` | `0` | Z-index for sorting within layer |
| `palette` | `string \| object` | — | Palette override (swap) |
| `variant` | `string` | — | Variant name from sprite's `variants` |
| `clip` | `string` | — | Active animation clip name |
| `attach` | `AttachRef` | — | Attach to parent item's attachment point |
| `emitter` | `EmitterConfig` | — | Particle emitter (see §10) |
| `tags` | `string[]` | — | Item tags |
| `if` | `string` | — | Conditional expression for runtime visibility |
| `groundY` | `number` | — | Y offset for shadow anchor |
| `meta` | `object` | — | Arbitrary metadata |

#### Scale chain

The effective scale for any item is: `canvas.scale × camera.zoom × ppuFactor × item.scale × sprite.baseScale` (where `ppuFactor = sprite.ppu / canvas.referencePpu` if `referencePpu` is set, else `1.0`)

This means:
- A house (`baseScale: 1.0`) at `scale: 0.4` appears pushed into the distance
- The same house at `scale: 2.0` fills the foreground
- `camera.zoom: 1.5` zooms the entire scene
- `canvas.scale: 2` doubles all pixels for display

#### Attachment references

An item can attach to another item's attachment point:

```json
{
  "name": "actors",
  "items": [
    { "asset": "hero_warrior", "x": 60, "y": 83, "id": "hero" },
    { "asset": "iron_sword", "attach": { "parent": "hero", "point": "hand_right" } }
  ]
}
```

The attached item's position is computed from the parent's position plus the attachment point offset. If the parent is animated and the attachment has per-frame positions (§5.9), the attached item moves with it.

| Field | Type | Description |
|-------|------|-------------|
| `parent` | `string` | ID of the parent item |
| `point` | `string` | Attachment point name on parent sprite |

Items that use `attach` do not need `x, y` — position is derived from the parent.

#### Shadow anchors

The optional `groundY` field on items specifies where a shadow ellipse should be drawn beneath a sprite. When sprites are scaled for depth (small = far away), the shadow scales proportionally and anchors the sprite visually to the ground plane:

```json
{ "asset": "flying_bird", "x": 100, "y": 30, "scale": 0.5, "groundY": 80 }
```

The renderer draws a shadow at `y: groundY` with size proportional to the item's effective scale.

## 10. Emitter Specification

Emitters define particle effects. An emitter spawns sprite instances at a configurable rate with randomized properties. The physics simulation is runtime — the format only stores the configuration.

### 10.1 Standalone emitter file

```json
{
  "format": "pixel-emitter-v1",
  "name": "campfire_sparks",
  "sprite": "spark",
  "ppu": 32,
  "rate": 5,
  "lifetime": { "min": 300, "max": 800 },
  "velocity": {
    "x": { "min": -1.5, "max": 1.5 },
    "y": { "min": -4.0, "max": -1.0 }
  },
  "gravity": 0.05,
  "fadeOut": true,
  "maxParticles": 20
}
```

### 10.2 Inline emitter on scene items

Emitter config can also be placed directly on a scene item:

```json
{
  "asset": "spark",
  "x": 84, "y": 72,
  "emitter": {
    "rate": 5,
    "lifetime": { "min": 300, "max": 800 },
    "velocity": {
      "x": { "min": -1.5, "max": 1.5 },
      "y": { "min": -4.0, "max": -1.0 }
    },
    "gravity": 0.05,
    "fadeOut": true,
    "maxParticles": 20
  }
}
```

### 10.3 Emitter fields

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `sprite` | `string` | — | Sprite to emit (required in standalone files; uses item's `asset` when inline) |
| `rate` | `number` | required | Particles spawned per second |
| `lifetime` | `{min, max}` | required | Particle lifetime range in ms |
| `velocity` | `{x: {min, max}, y: {min, max}}` | required | Initial velocity range per axis (pixels/second at native ppu) |
| `gravity` | `number` | `0` | Downward acceleration per frame (pixels/frame² at native ppu) |
| `fadeOut` | `boolean` | `false` | Fade opacity to 0 over lifetime |
| `scaleRange` | `{min, max}` | `{min: 1, max: 1}` | Random scale range per particle |
| `rotationSpeed` | `{min, max}` | `{min: 0, max: 0}` | Rotation speed range (degrees/second) |
| `maxParticles` | `integer` | `100` | Maximum simultaneous particles |
| `burst` | `integer` | — | If set, emit this many instantly then stop |
| `ppu` | `integer > 0` | `32` | Pixels Per Unit — anchors velocity and gravity to world space |

### 10.4 Emitter PPU

The `ppu` field on emitters serves the same purpose as on sprites and tilesets: it anchors spatial values (velocity, gravity) to world-space units.

**Formula:** At `ppu` 32, a velocity of `y: -4.0` means the particle moves 4 pixels per second upward — or `4 / 32 = 0.125` tiles per second.

When an emitter is placed in a scene with `canvas.referencePpu`, the renderer applies the same ppuFactor scaling:

`ppuFactor = emitter.ppu / canvas.referencePpu`

This ensures particle physics scale correctly when composed alongside sprites and tiles with different PPU values.

**Inline emitters** on scene items inherit context from the scene but still use their own `ppu` for spatial values. If `ppu` is omitted, it defaults to 32.

The renderer uses positional RNG seeded from scene state for deterministic particle output in multiplayer contexts.

## 11. Rendering Rules

### 11.1 Scaling

All pixel assets must be rendered with **nearest-neighbor scaling** (no smoothing, no anti-aliasing). This preserves the pixel art aesthetic at any zoom level.

In browser: `ctx.imageSmoothingEnabled = false`

### 11.2 Transparency

Transparent pixels must not overwrite destination pixels during compositing.

### 11.3 Clipping

Pixels outside the canvas bounds are clipped (not rendered).

### 11.4 Palette override

If a scene item provides `palette`, that palette replaces the sprite's default palette for rendering that item. This is the canonical mechanism for palette swaps (e.g., red potion → blue potion).

If a scene item provides `variant`, the variant's palette overrides are applied.

Priority: item `palette` > item `variant` palette > sprite default palette.

### 11.5 Scale chain

Effective pixel size = `canvas.scale × camera.zoom × ppuFactor × item.scale × sprite.baseScale`

Where `ppuFactor`:
- If `canvas.referencePpu` is set: `ppuFactor = sprite.ppu / canvas.referencePpu`
- Otherwise: `ppuFactor = 1.0`

This ensures assets with different `ppu` values render at correct world-space proportions when composed in a scene.

Integer scales produce perfect results. Fractional scales use nearest-neighbor sampling.

### 11.6 Emitter spatial scaling

When a scene has `canvas.referencePpu`, emitter spatial values (velocity, gravity) are scaled by the same ppuFactor:

`ppuFactor = emitter.ppu / canvas.referencePpu`

This ensures particle physics match the world scale regardless of the emitter's native PPU.

### 11.7 Blend mode compositing

Layer `blend` modes map to Canvas 2D `globalCompositeOperation`:

| Pixel Format | Canvas 2D |
|-------------|-----------|
| `normal` | `source-over` |
| `multiply` | `multiply` |
| `screen` | `screen` |
| `overlay` | `overlay` |
| `additive` | `lighter` |

Set the operation before drawing all items in that layer.

### 11.7 Animation

The renderer processes clips by:
1. Computing current time within the clip (accounting for `playback` mode)
2. For each track, finding the two surrounding keyframes
3. Interpolating the property value using the track's `easing`
4. Applying interpolated values to the sprite's render state

For `step` easing (used by `frame` tracks), the value jumps to the most recent keyframe's value with no interpolation.

For `pingpong` playback, the clip plays forward then backward, repeating. Duration covers one full forward pass.

### 11.8 Layer compositing order

1. Fill canvas with `background` color
2. For each layer (array order):
   a. Skip if `visible` is `false`
   b. Set layer `blend` mode
   c. Apply layer `opacity`
   d. Compute layer position (accounting for `scrollFactor` and camera)
   e. If `sort` is `"y"`, sort items by `y` position
   f. Render each item (accounting for `scale`, `rotation`, `flipX`, `flipY`, `opacity`)
3. Clip to canvas bounds

### 11.9 Recommended browser pipeline

1. Load palette registry
2. Load sprite and tileset registry
3. Parse and validate JSON files
4. Decode `raw` or `rle` rows into canonical row strings
5. Rasterize sprites to offscreen `<canvas>` elements at 1× scale
6. Cache as `ImageBitmap` for fast compositing
7. Compose scenes using `drawImage()` with `imageSmoothingEnabled = false`
8. Apply blend modes per layer via `globalCompositeOperation`
9. Scale final output to display size

No `.png` files on disk — `.pixel.json` is the source of truth. Pre-rendered PNGs are only produced as derived build artifacts (thumbnails, documentation, marketplace assets).

## 12. Validation Rules

A v1 validator should check:

### Sprite validation

- `format` is `"pixel-sprite-v1"`
- `width`, `height` are positive integers
- `encoding` is `"raw"` or `"rle"`
- If `pixels` present: row count = `height × (frameCount ?? 1)`; decoded width = `width`
- If `layers` present: each layer's row count and decoded width match
- Every palette key used in pixel data exists in the active palette
- `origin` coordinates are numeric if present
- `frameCount` is positive integer if present
- `ppu` is positive integer if present
- `baseScale` is positive number if present
- `regions` coordinates are within sprite bounds if present
- `attachments` coordinates are within sprite bounds if present

### Palette validation

- `format` is `"pixel-palette-v1"`
- All `entries` values are valid `PixelColor` strings
- All `aliases` keys exist in `entries`
- All `ramps` keys exist in `entries`
- Keys are single characters from the allowed set

### Tileset validation

- `format` is `"pixel-tileset-v1"`
- `tileWidth`, `tileHeight` are positive integers
- `ppu` is positive integer if present
- Every tile decodes to exactly tile size
- All palette keys exist in the tileset's palette

### Tilemap validation

- `format` is `"pixel-tilemap-v1"`
- `gridWidth`, `gridHeight` are positive integers
- `grid` row count = `gridHeight`; row length = `gridWidth`
- Every character in `grid` exists in `legend`
- Every `legend` value references a tile in the named tileset

### Scene validation

- `format` is `"pixel-scene-v1"`
- Canvas `width`, `height` are positive integers
- Canvas `referencePpu` is positive integer if present
- Layers array exists and is non-empty
- Every item has `asset` and either (`x`, `y`) or `attach`
- Palette overrides are valid if inline
- Referenced clips exist on the target sprite
- Referenced attachment points exist on the target sprite
- Blend mode is one of the allowed values

### Emitter validation

- `format` is `"pixel-emitter-v1"`
- `rate` is positive number
- `lifetime.min` ≤ `lifetime.max`
- `velocity` ranges are valid numbers
- `maxParticles` is positive integer if present
- `ppu` is positive integer if present

## 14. RLE Decoder Reference

```ts
export function decodeRleRow(row: string): string {
  const tokens = row.trim().split(/\s+/).filter(Boolean);
  let out = '';

  for (const token of tokens) {
    const match = token.match(/^(\d+)(.)$/);
    if (!match) {
      throw new Error(`Invalid RLE token: ${token}`);
    }

    const count = Number(match[1]);
    const symbol = match[2];

    if (!Number.isInteger(count) || count <= 0) {
      throw new Error(`Invalid RLE count in token: ${token}`);
    }

    out += symbol.repeat(count);
  }

  return out;
}

export function decodePixelRows(
  encoding: 'raw' | 'rle',
  rows: string[],
  width: number,
  expectedRows: number
): string[] {
  if (rows.length !== expectedRows) {
    throw new Error(`Expected ${expectedRows} rows, got ${rows.length}`);
  }

  const decoded = rows.map((row) =>
    encoding === 'raw' ? row : decodeRleRow(row)
  );

  for (const row of decoded) {
    if (row.length !== width) {
      throw new Error(`Decoded row length ${row.length} does not match width ${width}`);
    }
  }

  return decoded;
}
```

## 15. Best Practices

### For LLM generation

- **Use semantic palette aliases** — "bark-dark" is easier to reason about than "#2b1d0e"
- **Use color ramps** — give the LLM a shading vocabulary
- **Set constraints** — `maxColors: 6` prevents palette explosion
- **Use symmetry hints** — halves token cost for symmetrical sprites
- **Use regions** — enables partial regeneration ("redo just the head")
- **Provide generation metadata** — tracks lineage for iterative refinement
- **Keep sprites small** — 8×8 to 32×32 is the sweet spot for LLM generation

### For encoding

- **Use raw** for small sprites (≤16×16) and hand-edited files
- **Use RLE** for large sprites, terrain masks, weather overlays, and sparse regions

### For scenes

- **Use y-sort layers** for objects and actors that overlap dynamically
- **Use parallax** for depth on scrolling scenes
- **Use blend modes** for lighting effects — additive for glow, multiply for shadows
- **Use attachment points** instead of hardcoded positions for equipment/accessories
- **Use variants** for palette swaps instead of duplicating sprites

### For tilemaps

- **Use tile connectivity** for auto-tiling paths, rivers, and walls
- **Use tile transforms** (flip/rotate) to reduce tile count

## 16. Canonical Limitations of v1

These are intentionally excluded from v1:

- **Multi-character palette keys** — breaks the char-grid model that makes text-as-art work
- **Binary serialization** — text IS the feature
- **Inverse kinematics / skeletal chains** — attachment points cover v1 composition needs
- **Shader code** — runtime concern, not file format
- **Per-pixel normal maps** — deferred to v2 (requires WebGL)
- **SVG or vector paths** — different paradigm
- **Procedural generation rules** — constraints are advisory hints, not executable programs
- **Layer grouping** — flat layer lists are simpler and sufficient
- **Onion skinning** — editor feature, not file format
- **Undo history** — editor state, not interchange format

## 17. Summary

Pixel Format v1 is the first pixel art format designed for LLM generation. It is:

- **Text-native** — the raw text IS the art; LLMs read, write, and transform sprites as character grids
- **JSON-based** — parses natively in JavaScript/TypeScript
- **Palette-constrained** — semantic color vocabularies guide generation quality
- **Composable** — sprites, layers, scenes, tilemaps, and attachments combine cleanly
- **Animated** — keyframe clips with frame sequences and property interpolation
- **Depth-aware** — per-item scaling, parallax layers, y-sort, and camera control
- **Deterministic** — same file always renders the same image
- **Git-diffable** — every change is a meaningful text diff
- **Extensible** — all new fields are optional; old files always parse correctly

It sits between the LLM-generated narrative description and the rendered pixel-art output — a stable visual asset language that is moddable, composable, and multiplayer-safe.
