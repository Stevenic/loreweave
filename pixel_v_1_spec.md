# Pixel Format v1 Specification

## 1. Overview

`Pixel Format v1` is a deterministic, human-readable asset and scene format for retro pixel-art rendering in JavaScript and TypeScript game engines.

It is designed for:

- reusable sprite assets
- strict palette-based rendering
- layered scene composition
- deterministic visual output
- easy git diffing and hand editing
- future compatibility with binary or compressed encodings

The format is intended for multiplayer persistent worlds where the visual layer should be stable, composable, and renderer-controlled rather than regenerated by an image model each time.

---

## 2. Design Goals

The format should be:

1. **Deterministic** — the same file always renders the same image.
2. **Composable** — sprites, tiles, and scenes can be combined.
3. **Palette-locked** — assets render from a known palette.
4. **Readable** — easy to inspect and edit in source control.
5. **Extensible** — supports future animation, lighting, metadata, and binary packing.
6. **JavaScript-friendly** — parses natively as JSON.

---

## 3. File Types

Pixel v1 defines three primary JSON-based file categories.

### 3.1 Sprite files
Suggested extension:

- `.pixel.json`

Used for single reusable assets such as:

- character sprites
- trees
- rocks
- doors
- campfires
- icons

### 3.2 Tileset files
Suggested extension:

- `.tileset.pixel.json`

Used for named collections of tile-sized assets, often terrain or wall pieces.

### 3.3 Scene files
Suggested extension:

- `.scene.pixel.json`

Used for composition of multiple assets onto a canvas.

---

## 4. Top-Level Format Versions

Every file must declare a `format` string.

Allowed v1 values:

- `pixel-sprite-v1`
- `pixel-tileset-v1`
- `pixel-scene-v1`
- `pixel-palette-v1`

Renderers must reject unknown `format` values unless explicitly configured for forward compatibility.

---

## 5. Coordinate System

All coordinates use a top-left origin.

- `x` increases left to right
- `y` increases top to bottom

Sprites may define an `origin` anchor point for placement in scenes.

Example:

- bottom-center origin for characters and trees
- bottom-middle of a building footprint

---

## 6. Palette Specification

A palette maps symbol keys to colors.

### 6.1 Supported palette value types

Palette entries must be one of:

- `"transparent"`
- `"#RRGGBB"`
- `"#RRGGBBAA"`

Recommended v1 renderer behavior:

- `transparent` → RGBA `(0, 0, 0, 0)`
- `#RRGGBB` → alpha defaults to `FF`
- `#RRGGBBAA` → explicit alpha

### 6.2 Allowed symbol keys

In v1, palette keys are single characters for readability.

Recommended allowed keys:

- digits: `0-9`
- uppercase letters: `A-Z`
- lowercase letters: `a-z`
- optional symbols: `! @ # $ % & * + = ?`

A renderer may support a narrower subset, but all rows in the image data must use keys defined in the palette.

### 6.3 Example palette file

```json
{
  "format": "pixel-palette-v1",
  "name": "fantasy16",
  "entries": {
    "0": "transparent",
    "1": "#2b1d0e",
    "2": "#5a3a22",
    "3": "#8b5a2b",
    "4": "#e2a13b",
    "5": "#ffd46b",
    "6": "#254d2a",
    "7": "#3a6b3a",
    "8": "#5b8f4c",
    "9": "#91b86f",
    "A": "#6d7a8a",
    "B": "#a7b5c5",
    "C": "#d8e2f1",
    "D": "#7f2f35",
    "E": "#bf5a5a",
    "F": "#10151f"
  }
}
```

## 7. Sprite File Specification

### 7.1 Required fields

```json
{
  "format": "pixel-sprite-v1",
  "name": "campfire_small",
  "width": 8,
  "height": 8,
  "palette": "fantasy16",
  "encoding": "raw",
  "pixels": [
    "00044000",
    "00045500",
    "00044500",
    "00035300",
    "00313130",
    "00111110",
    "000AA000",
    "00000000"
  ]
}
```

Required fields:

- `format: string`
- `name: string`
- `width: integer > 0`
- `height: integer > 0`
- `palette: string | object`
- `encoding: "raw" | "rle"`
- `pixels: array`

### 7.2 Optional fields

- `origin: { x: number, y: number }`
- `tags: string[]`
- `collision: { x, y, width, height }`
- `variants: string[]`
- `meta: object`

### 7.3 Raw encoding

When `encoding` is `raw`, `pixels` must be an array of strings.

Rules:

- array length must equal `height`
- each row length must equal `width`
- each character in a row is one palette key
- every palette key must exist in the active palette

Example:

```json
{
  "format": "pixel-sprite-v1",
  "name": "smoke_plume",
  "width": 6,
  "height": 4,
  "palette": {
    "0": "transparent",
    "A": "#a7b5c5",
    "B": "#d8e2f1"
  },
  "encoding": "raw",
  "pixels": [
    "00BB00",
    "0BBBB0",
    "00BB00",
    "000000"
  ]
}
```

## 8. RLE Encoding

Pixel v1 includes first-class RLE support to reduce size while keeping files readable.

### 8.1 Purpose

RLE is recommended when:

- assets contain repeated runs of the same color
- large tiles or overlays have long flat regions
- scene masks or weather layers would otherwise be verbose

### 8.2 Encoding model

When `encoding` is `rle`, `pixels` must be an array of row strings, where each row is composed of repeated **run tokens**.

A run token format is:

```text
<count><symbol>
```

Examples:

- `8A` = eight pixels of palette key `A`
- `30` = three pixels of transparent key `0`
- `12B` = twelve pixels of palette key `B`

Multiple run tokens are concatenated in one row using spaces for readability.

Example row:

```text
3A 2B 3A
```

means:

```text
AAABBAAA
```

### 8.3 RLE row rules

For each row:

- tokens must parse into a total decoded width equal to `width`
- row count must still equal `height`
- symbols must exist in the palette
- counts must be positive integers
- zero-length runs are invalid

### 8.4 RLE example

```json
{
  "format": "pixel-sprite-v1",
  "name": "fog_band",
  "width": 16,
  "height": 4,
  "palette": {
    "0": "transparent",
    "B": "#d8e2f1"
  },
  "encoding": "rle",
  "pixels": [
    "4B 4B 4B 4B",
    "2B 12B 2B",
    "1B 14B 1B",
    "16B"
  ]
}
```

### 8.5 Canonical decoding algorithm

A v1-compliant decoder should:

1. split each row by whitespace
2. parse each token as `/^(\d+)(.)$/`
3. expand the symbol by the given count
4. concatenate runs into one decoded row string
5. verify final row length equals `width`

### 8.6 Canonical encoding recommendation

For consistent diffs, writers should:

- merge adjacent runs of the same symbol
- use a single space between run tokens
- avoid emitting `1A 1A`; emit `2A` instead

### 8.7 Example raw vs RLE equivalence

Raw:

```json
{
  "width": 8,
  "height": 2,
  "encoding": "raw",
  "pixels": [
    "000AA000",
    "00AAAA00"
  ]
}
```

RLE equivalent:

```json
{
  "width": 8,
  "height": 2,
  "encoding": "rle",
  "pixels": [
    "30 2A 30",
    "20 4A 20"
  ]
}
```

## 9. Palette Resolution Rules

A sprite `palette` field may be either:

1. a string palette name, resolved by the asset pipeline
2. an inline palette object

If a string name is used, the runtime must resolve it from a palette registry.

If resolution fails, rendering must fail with an explicit error.

---

## 10. Sprite Validation Rules

A valid sprite must satisfy all of the following:

- `width` and `height` are positive integers
- `encoding` is supported
- `pixels` has exactly `height` rows
- each decoded row length is exactly `width`
- every symbol used appears in the palette
- `origin`, if present, is numeric

---

## 11. Tileset File Specification

Tilesets organize multiple small assets under a common palette.

### 11.1 Example

```json
{
  "format": "pixel-tileset-v1",
  "name": "forest_ground_tiles",
  "tileWidth": 8,
  "tileHeight": 8,
  "palette": "fantasy16",
  "tiles": {
    "grass": {
      "encoding": "raw",
      "pixels": [
        "77777777",
        "78878787",
        "77788777",
        "78877787",
        "77777777",
        "78788787",
        "77777777",
        "77777777"
      ]
    },
    "path": {
      "encoding": "rle",
      "pixels": [
        "83",
        "23 24 23 24",
        "84",
        "24 23 24 23",
        "83",
        "84",
        "83",
        "84"
      ]
    }
  }
}
```

### 11.2 Required fields

- `format`
- `name`
- `tileWidth`
- `tileHeight`
- `palette`
- `tiles`

Each tile entry must provide:

- `encoding`
- `pixels`

Each tile uses `tileWidth` and `tileHeight` as its dimensions.

## 12. Scene File Specification

Scene files place assets on a canvas.

### 12.1 Example

```json
{
  "format": "pixel-scene-v1",
  "name": "forest_clearing_camp",
  "canvas": {
    "width": 160,
    "height": 96,
    "background": "#89b9ff"
  },
  "layers": [
    {
      "name": "world",
      "items": [
        { "asset": "pine_tree_tall", "x": 18, "y": 70 },
        { "asset": "pine_tree_tall", "x": 38, "y": 66 },
        { "asset": "campfire_small", "x": 82, "y": 76 },
        { "asset": "smoke_plume", "x": 83, "y": 58 }
      ]
    },
    {
      "name": "actors",
      "items": [
        { "asset": "hero_ranger_front", "x": 60, "y": 83 },
        { "asset": "hero_cleric_front", "x": 74, "y": 84 }
      ]
    }
  ]
}
```

### 12.2 Required fields

- `format`
- `name`
- `canvas`
- `layers`

### 12.3 Canvas fields

Required:

- `width`
- `height`

Optional:

- `background`
- `scale`
- `meta`

### 12.4 Layer fields

Required:

- `name`
- `items`

### 12.5 Item fields

Required:

- `asset`
- `x`
- `y`

Optional:

- `palette`
- `opacity`
- `flipX`
- `flipY`
- `rotation` (optional, though many retro renderers may omit support)
- `z`
- `variant`
- `tags`
- `if`
- `meta`

### 12.6 Scene composition semantics

The renderer must process layers in array order.
Within a layer, items should render in listed order unless the engine sorts by explicit `z` or another documented rule.

Recommended default for v1:

1. render layers in file order
2. render items in file order inside each layer
3. optionally allow stable sort by `z`

---

## 13. Scene Asset Resolution

The `asset` field must resolve to a sprite or tileset member known to the runtime.

Resolution strategy is implementation-defined, but recommended options include:

- asset registry by name
- relative file path
- content-addressable ID

For v1, the simplest recommendation is a runtime asset registry keyed by unique asset names.

---

## 14. Rendering Rules

### 14.1 Scaling

Pixel v1 assets should be rendered with nearest-neighbor scaling only.

No smoothing should be applied.

### 14.2 Transparency

Transparent pixels must not overwrite destination pixels.

### 14.3 Out-of-bounds pixels

Pixels outside the canvas should be clipped.

### 14.4 Palette override

If a scene item provides `palette`, that palette replaces the sprite’s default palette for rendering that item.

This is the canonical mechanism for palette swaps.

## 16. RLE TypeScript Decoder Reference

```ts
export function decodeRleRow(row: string): string {
  const tokens = row.trim().split(/\s+/).filter(Boolean);
  let out = '';

  for (const token of tokens) {
    const match = token.match(/^(\d+)(.)$/);
    if (!match) {
      throw new Error(`Invalid RLE token: ${token}`);
    }

    const count = Number(match[1]);
    const symbol = match[2];

    if (!Number.isInteger(count) || count <= 0) {
      throw new Error(`Invalid RLE count in token: ${token}`);
    }

    out += symbol.repeat(count);
  }

  return out;
}
```

### 16.1 Whole sprite decode helper

```ts
export function decodePixelRows(
  encoding: 'raw' | 'rle',
  rows: string[],
  width: number,
  height: number
): string[] {
  if (rows.length !== height) {
    throw new Error(`Expected ${height} rows, got ${rows.length}`);
  }

  const decoded = rows.map((row) =>
    encoding === 'raw' ? row : decodeRleRow(row)
  );

  for (const row of decoded) {
    if (row.length !== width) {
      throw new Error(`Decoded row length ${row.length} does not match width ${width}`);
    }
  }

  return decoded;
}
```

## 17. Validation Recommendations

A v1 validator should check:

### Sprite validation

- legal `format`
- `width`, `height` positive integers
- supported `encoding`
- row count and decoded widths
- palette key existence
- duplicate palette entries are allowed, duplicate keys are not

### Tileset validation

- legal `format`
- positive `tileWidth`, `tileHeight`
- every tile decodes to exactly tile size

### Scene validation

- legal `format`
- positive canvas size
- layers array exists
- every item has asset and coordinates
- optional palette override is valid if inline

---

## 18. Recommended Runtime Pipeline

1. Load palette registry
2. Load sprite and tileset registry
3. Parse JSON into typed objects
4. Validate objects
5. Decode `raw` or `rle` rows into canonical row strings
6. Convert rows to pixel buffers or canvases
7. Compose scenes onto a destination canvas
8. Export to PNG if needed

---

## 19. Canonical Limitations of v1

Pixel v1 intentionally does not standardize all future features.
These are left implementation-defined or deferred to v2:

- animation timelines
- skeletal composition
- shader effects
- lighting models
- per-pixel normal maps
- procedural tileset autotiling
- binary serialization
- multi-character palette keys

---

## 20. Recommended Future Extensions

For v2 or optional extensions, consider adding:

- animation strips
- tilemaps with tile indices
- named layers with blend modes
- light and shadow masks
- asset references by URI
- binary-packed `.pixelb` files
- optional gzip support in tooling
- metadata for authoring provenance

---

## 21. Best Practices

### Use raw encoding when:

- sprites are small
- readability matters most
- artists are hand-editing files

### Use RLE when:

- overlays are large
- rows contain long repeated runs
- terrain masks or weather effects are sparse or flat

### Use scene files when:

- visuals are built from reusable pieces
- world state determines composition
- you want deterministic output for multiplayer

## 23. Summary

Pixel Format v1 provides a clean baseline for retro game visuals:

- JSON-based
- strict palette control
- raw and RLE encoding
- reusable sprites and tilesets
- deterministic scene composition
- easy integration with JavaScript and TypeScript

For your game, this gives you a stable visual asset language that can sit between:

- the LLM-generated narrative description
- and the actual rendered pixel-art output

That makes the rendering layer consistent, moddable, and multiplayer-safe.
