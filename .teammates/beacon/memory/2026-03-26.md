## Task 29: Fix Client-Side RLE Decoder

- Root cause: client-side `decodeRleRow()` parsed character-by-character instead of splitting by whitespace first
- The RLE format uses space-delimited tokens like `"640 24 620"` where each token is `<count><key>` (e.g., "640" = 64 repetitions of palette key '0')
- Old decoder saw "640" as numStr="640" then used the space as the palette key — completely wrong parsing
- Server-side decoder (`packages/pixel/src/decoder.ts`) correctly splits by whitespace, then regex matches `^(\d+)(.)$` per token
- Rewrote client-side decoder to match: `row.trim().split(/\s+/)` then regex match each token
- All 8 workspaces build clean
- Files changed: `services/pixel-explorer/src/html.ts`

## Task 30: Fix Regex Escaping in Template Literal — RLE Still Broken

- Root cause: `decodeRleRow()` JavaScript code lives inside a TypeScript template literal (backtick string) in `html.ts`
- Regex `/\s+/` and `/^(\d+)(.)$/` lost their backslashes during template literal evaluation — `\s` → `s`, `\d` → `d`
- Browser received `/s+/` (splits on literal 's') and `/^(d+)(.)$/` (matches literal 'd') — completely wrong
- Token splitting never worked (no 's' characters in RLE data), so decoded rows were empty strings → nothing drawn
- Fix: double-escape all regex character classes — `\\s` and `\\d` in the TypeScript source
- This was the actual root cause across Tasks 28-30; the algorithm was correct but the escaping was not
- Files changed: `services/pixel-explorer/src/html.ts` (lines 626, 630)
- All 8 workspaces build clean
- Key: ANY regex with `\s`, `\d`, `\w`, `\b`, etc. inside a JS template literal needs `\\s`, `\\d`, etc.
