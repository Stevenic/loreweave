# Scribe — Daily Log — 2026-03-27

## Task 1: NPC System Design Proposal

- **What:** Designed the comprehensive NPC system architecture for LoreWeave — how NPCs spawn, persist, behave, and interact with players.
- **Context:** stevenic asked "how do we think NPCs work?" Current state has three disconnected layers: game types (Character/EntityRef — thin), world gen spawn rules (creatures only), and archetype templates (rich but no runtime model). The gap is connecting archetypes to living NPCs.
- **Key design decisions proposed:**
  - Three-tier model: Creatures (ephemeral, spawn rules), Residents (persistent, generated from archetypes), Named Characters (persistent, hand-authored by Bard)
  - `NPCRecord` type — persistent data with identity, traits, voice pattern, knowledge, disposition, schedule
  - LLM-generated dialogue — no dialogue trees, no stored lines. LLM gets NPC personality + knowledge + disposition + context and generates naturally.
  - Schedule-driven location — NPCs move between locations by time of day, creating natural scene variety
  - Disposition system — -100 to +100 toward player, gates trading, quests, and secret sharing
  - Skill-gated knowledge — Persuasion/Insight checks against DCs based on knowledge tier (always/sometimes/rarely)
  - NPC lifecycle: generated at settlement creation, persist forever, can die and be replaced via world healing rules
- **Open questions posed to stevenic:**
  1. NPC memory of past player interactions
  2. NPC-to-NPC relationships
  3. Companion NPCs (v1 or v2?)
  4. NPC count limits per settlement
  5. Trading system depth
- **Spec changes proposed (pending approval):** New `NPCRecord` type, new §8A in world gen spec (settlement NPC generation), expanded §4.4.3 (NPC context in narrative), new NPC system section in loreweave spec
- **Files changed:** None (proposal only — awaiting stevenic decisions)

## Task 2: Engine Documentation

- **What:** Created the `docs/` directory with 4 technical documents covering the full engine architecture — how generation works, how the narrative pipeline processes turns, how the rules engine implements D&D 5e, and how world configuration works.
- **Context:** stevenic asked to document how the engine works. No `docs/` directory existed. All knowledge was spread across specs and source code.
- **Approach:** Analyzed all source files across all 6 packages (types, world, rules, narrative, agents, pixel) to document what's actually implemented, not just what's spec'd.
- **Documents created:**
  - `docs/engine-overview.md` — Package architecture, single-turn data flow diagram, 6 core design properties (deterministic, pure functions, lazy generation, no cross-chunk deps, world-agnostic, LLM as renderer)
  - `docs/world-generation.md` — Full 7-stage chunk pipeline (biomes → elevation → surface → resources → structures → features → spawns), tile structure, 3 randomness algorithms (OpenSimplex2, FNV-1a, xorshift32), RNG stream isolation, World class API, unbuilt systems noted
  - `docs/rules-engine.md` — Dice system, ability checks/skill checks/saving throws, combat (initiative, melee/ranged attacks, damage), NPC generation from archetypes, stat tiers, encounter system, ward/fray/skill challenge mechanics
  - `docs/narrative-engine.md` — DungeonMaster turn pipeline, intent parser (10 action types with keyword tables), action resolver (per-action mechanics), context assembler (6 components), prompt builder (system + user prompts), vocabulary injection, WorldConfig, GameSession, NarrativeAdapter interface
  - `docs/world-configuration.md` — Directory structure for worlds/shared/ and worlds/<name>/, config file reference (persona, stat-tiers, ward, fray, encounters, creatures, skill-challenges), archetype system (location + NPC archetypes with probability tiers), vocabulary tables, how to add a new world
- **Key decisions:**
  - Documented from source code, not specs — these docs reflect what's built
  - Noted unbuilt systems (location generator, weave state overlay, additional biomes) as "not yet built"
  - Included concrete code examples and data structures throughout
  - Organized as standalone docs that can be read independently but cross-reference each other
- **Files created:** `docs/engine-overview.md`, `docs/world-generation.md`, `docs/rules-engine.md`, `docs/narrative-engine.md`, `docs/world-configuration.md`
