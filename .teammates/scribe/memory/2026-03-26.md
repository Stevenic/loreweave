## Task 3: Full Pixel Format v1 Revision Proposal — LLM-Native Format

- **What:** Delivered comprehensive proposal to transform Pixel Format v1 into the first pixel art format designed for LLM generation. 18 features across 4 categories.
- **Context:** stevenic said "go for everything out the gate" on depth features AND wants to discuss v2 ideas. Vision: "a killer new pixel art format optimized for generation by LLMs."

  - **LLM Generation (8):** semantic palette aliases, color ramps, symmetry hints (horizontal/vertical/both with half-data generation), sprite templates with named regions, generation constraints (`maxColors`, `outline`, `noIslands`), variant rules (palette swaps, scale, row removal), tile connectivity (Wang tile edge matching), generation metadata (prompt, model, seed, lineage)

- **Key design insight:** The single-char grid model is the core LLM optimization. Everything builds on keeping that simple. Symmetry hints cut token count in half. Templates give LLMs structural guidance. Ramps give LLMs a vocabulary for shading.

## Task 5: Definitive Sprite Format Design — LLM-Optimized

- **What:** Designed the complete, definitive sprite format for Pixel Format v1, answering stevenic's question: "what does an individual sprite look like that an LLM can generate and transform?"
- **Context:** stevenic wants to nail down the exact format. Agrees on avoiding binary, wants RLE support. This is the atomic unit — everything else builds on it.
- **Key design decisions:**
  - **Symmetry hints** (`horizontal`/`vertical`/`both`/`none`) — LLM generates only half the pixel grid, renderer mirrors. 50% token reduction for symmetrical assets.
  - **Animation via vertical stacking** — frames stacked in `pixels` array, `height` = frame height, total rows = height × frameCount. Clips with named tags and playback modes (loop/once/pingpong).
  - **Regions** (named rectangles) — enables partial regeneration ("just redo the head"), attachment points, multiple hitboxes.
  - **Constraints** — `maxColors`, `outline`, `noIslands` — generation quality hints for LLMs.
  - **Generation metadata** — prompt, model, seed, parent lineage for variant tracking.
  - **Variant rules** (declarative) — palette swaps and scale overrides without duplicating pixel data.
  - **Palette enhancements** — semantic aliases (char→name mapping) and color ramps (ordered gradients) in palette files.
  - **baseScale** on sprites for scene depth composition.
  - All new fields are optional — format is fully backwards compatible with existing sprites.
  - Deliberately excluded from sprites: internal layers, multi-char keys, binary encoding, skeletal animation, blend modes, procedural rules.
- **Open decision points posed to stevenic:**
  1. Symmetry: store half pixels or full pixels with hint?
  2. Variants: simple (palette+scale) or complex (row removal, region replacement)?
  3. Animation: vertical stacking in one array or structured array-of-arrays?
  4. Constraints: hard reject or advisory warn?
- **Files changed:** None (design proposal only — awaiting stevenic decisions before spec edit)

## Task 6: Rendering Pipeline Design — .pixel vs .png

- **What:** Answered stevenic's question about whether sprites should have both `.pixel.json` source and `.png` pre-rendered files, scaling capabilities, and lighting/overlay composition. Designed the two-tier rendering model.
- **Context:** stevenic wants browser-based JS rendering. Asked about performance, rescaling, and overlaying PNGs with lighting effects.
- **Key design decisions:**
  - **No `.png` on disk** — `.pixel.json` is source-of-truth. Rasterize to offscreen `<canvas>` at load time, cache as `ImageBitmap`. No drift risk, no build step friction.
  - **Scaling** — `ctx.drawImage()` with `imageSmoothingEnabled = false` (nearest-neighbor). Scale chain: `canvas.scale × layer.scrollFactor × item.scale × sprite.baseScale`.
  - **Lighting/effects** — Canvas 2D blend modes (`multiply`, `lighter`, `screen`, `overlay`) handle tinting, additive lighting, glow, shadows, and masking natively. No WebGL needed for v1.
  - **Lighting assets are just sprites** — a warm light overlay is a sprite with semi-transparent palette colors, composed via blend modes at the scene layer level.
  - **WebGL deferred** — only needed for per-pixel normal map lighting, shader-based palette swaps, particles, or post-processing. Canvas 2D covers v1 needs.
  - **`.png` export** — only as a derived build artifact (thumbnails, marketplace, docs), never as source.
- **Spec implications:** `blend` mode on scene layers (already in 21-feature proposal) is the key addition. No new file types needed.
- **Files changed:** None (design discussion only)
- **Still open:** 4 sprite format decisions from Task 5, plus whether to write a "Rendering Pipeline" spec section

## Task 12: PPU (Pixels Per Unit) Proposal

- **What:** Analyzed stevenic's follow-up question about DPI/resolution for the pixel format. Proposed `ppu` (Pixels Per Unit) as a sprite-level field — the "DPI for a tile-based world."
- **Context:** stevenic pointed out that even with size classes, the format has no way to express what "1x" means. Scale chain exists but has no anchor to world space. An LLM can't reason about relative proportions without it.
- **Key proposal:**
  - `ppu` = Pixels Per Unit, where unit = 1 world tile
  - Formula: `world size (tiles) = pixel dimensions ÷ ppu`
  - Default: 32 (characters/items). Buildings at 16, icons at 64, terrain at 8.
  - Sprite-level field (self-describing, LLM-friendly)
  - Optional scene-level `referencePpu` for auto-scaling
  - Avoids "DPI" terminology since we're not dealing with print/inches
- **Spec changes needed:** Add `ppu` to PixelSprite and PixelTileset, optionally `referencePpu` to scene canvas. Update schema, LLM guide, and TypeScript types.
- **Status:** Awaiting stevenic approval before spec edit
- **Files changed:** None (proposal only)
