# LoreWeave — World Generation Specification (v1)

*Inspired by Minecraft's world generation architecture, adapted for LoreWeave's 2D top-down tile-based world.*

---

## 1. Overview

World generation in LoreWeave converts a **seed** (64-bit integer) into a deterministic, infinite 2D tile grid organized into chunks. The system uses layered noise functions to produce terrain that is locally coherent, globally varied, and biome-driven.

### Design Goals

- **Deterministic**: Given the same seed and chunk coordinates, generation always produces identical output
- **Lazy**: Chunks generate on demand, not upfront
- **Composable**: Each generation stage is a pure function that transforms the chunk state
- **Scenario-aware**: Procedural output can be overridden locally by scenario injections (see `loreweave_spec.md` §4.6)

### Relationship to Core Spec

This spec expands `loreweave_spec.md` §4.1 (World Engine). The `WorldTile` and `Chunk` types defined there are the output of this generation pipeline. This spec defines *how* those types get populated.

## 2. Seeds and Randomness

### 2.1 World Seed

Every world has a single **world seed** — a 64-bit integer provided at world creation. All generation derives from this seed.

```ts
type WorldSeed = bigint; // 64-bit
```

If the player provides a string instead of a number, hash it to a 64-bit integer (e.g., via FNV-1a or similar).

### 2.2 Positional RNG

For any operation that needs randomness at a specific location, derive a **positional seed** by combining the world seed with the coordinates:

```
positionalSeed(worldSeed, x, y) = hash(worldSeed, x, y)
```

This ensures:
- Same position always produces the same random sequence
- Different positions produce uncorrelated sequences
- Generation order does not affect output (unlike Minecraft, where chunk order can cause non-determinism)

### 2.3 Noise Functions

All terrain generation uses **gradient noise** (Perlin or Simplex). Noise functions are seeded from the world seed with a domain-specific offset so each noise layer is independent.

Key properties:
- Output range: `[-1.0, 1.0]`
- Continuous: nearby inputs produce nearby outputs
- Deterministic: same input always produces same output

**Octave noise** combines multiple noise samples at different frequencies and amplitudes to create natural-looking variation:

```
octaveNoise(x, y, octaves, persistence) =
  Σ (amplitude_i × noise(x × frequency_i, y × frequency_i))
  where frequency_i = 2^i, amplitude_i = persistence^i
```

## 3. Chunk System

### 3.1 Chunk Dimensions

Chunks are **32×32 tiles**. This balances generation cost against granularity.

```ts
const CHUNK_SIZE = 32;

type ChunkCoord = {
  cx: number; // chunk X (world X = cx * CHUNK_SIZE + localX)
  cy: number; // chunk Y (world Y = cy * CHUNK_SIZE + localY)
};
```

### 3.2 Chunk Lifecycle

Chunks progress through generation stages in order. A chunk may be **frozen** at any stage if no player is nearby, and resumed later.

```
empty → biomes → elevation → surface → structures → features → spawns → complete
```

| Stage | Description |
|-------|-------------|
| `empty` | Chunk allocated, no data |
| `biomes` | Biome assigned to each tile |
| `elevation` | Elevation values computed |
| `surface` | Surface type and resources determined per biome + elevation |
| `structures` | Structures placed (or reserved from neighbor structure starts) |
| `features` | Decorative features placed (trees, rocks, flowers, etc.) |
| `spawns` | NPC/creature spawn points determined |
| `complete` | Fully generated, ready for gameplay |

### 3.3 Neighbor Dependency

Some stages (structures, features) may read data from adjacent chunks. When a chunk at stage `S` needs neighbor data, those neighbors must be at stage `S-1` or later. The generator must handle this by generating neighbors to the required stage first.

## 4. Biome Generation

### 4.1 Noise Parameters

Biomes are determined by **three noise parameters** evaluated at each tile position. This is a simplified adaptation of Minecraft's 6-parameter system, reduced to 3 dimensions because LoreWeave's 2D world doesn't need depth or the full continentalness range.

| Parameter | Range | What It Controls |
|-----------|-------|-----------------|
| **Temperature** | `[-1.0, 1.0]` | Hot vs. cold climate |
| **Moisture** | `[-1.0, 1.0]` | Wet vs. dry conditions |
| **Elevation Noise** | `[-1.0, 1.0]` | Low-lying vs. highland terrain (also feeds into §5) |

Each parameter is generated by an independent octave noise function:

```ts
temperature(x, y)     = octaveNoise(x, y, octaves: 4, persistence: 0.5, seed: worldSeed + 0)
moisture(x, y)        = octaveNoise(x, y, octaves: 4, persistence: 0.5, seed: worldSeed + 1)
elevationNoise(x, y)  = octaveNoise(x, y, octaves: 6, persistence: 0.5, seed: worldSeed + 2)
```

**Frequency scale**: Base frequency should produce biome regions of roughly 100–200 tiles across. Tune the input scaling factor to achieve this (e.g., `noise(x * 0.005, y * 0.005)`).

### 4.2 Biome Selection Table

Each biome occupies a region in 3D parameter space. To determine the biome at a tile, find the biome whose center is closest in Euclidean distance to the tile's `(temperature, moisture, elevationNoise)` values. This guarantees every tile maps to exactly one biome, with no gaps.

| Biome | Temperature | Moisture | Elevation Noise | Description |
|-------|-------------|----------|-----------------|-------------|
| **Forest** | `0.0 to 0.5` | `0.2 to 0.8` | `-0.3 to 0.3` | Temperate woodland. Dense trees, moderate resources. |
| **Plains** | `0.0 to 0.6` | `-0.5 to 0.2` | `-0.5 to 0.0` | Open grassland. Few trees, good visibility. |
| **Mountain** | `-0.3 to 0.3` | `-0.3 to 0.3` | `0.5 to 1.0` | High elevation. Rocky, sparse vegetation. |
| **Swamp** | `0.1 to 0.6` | `0.5 to 1.0` | `-1.0 to -0.3` | Low-lying wetland. Dense foliage, murky water. |
| **Tundra** | `-1.0 to -0.3` | `-0.8 to 0.2` | `-0.2 to 0.5` | Frozen terrain. Snow, ice, minimal life. |
| **Desert** | `0.6 to 1.0` | `-1.0 to -0.3` | `-0.3 to 0.3` | Arid sand. No trees, scarce resources. |
| **Taiga** | `-0.5 to 0.0` | `0.2 to 0.7` | `0.0 to 0.5` | Cold coniferous forest. Pine trees, wolves. |

**Closest-biome fallback**: If a tile's parameters don't fall within any biome's defined ranges, assign the biome whose center point `(avg(temp), avg(moisture), avg(elevation))` is nearest in 3D Euclidean distance. This mirrors Minecraft's approach and guarantees no tile is unassigned.

### 4.3 Biome Blending

At biome boundaries, tiles within a **4-tile blend radius** sample biomes from a 9×9 neighborhood and use the most common result. This prevents harsh single-tile biome transitions and creates natural-looking edges.

### 4.4 River Generation

Rivers are generated as a separate overlay using a **ridged noise** function:

```ts
riverNoise(x, y) = 1.0 - abs(octaveNoise(x, y, octaves: 3, persistence: 0.5, seed: worldSeed + 3))
```

Where `riverNoise > 0.92`, the tile is marked as a **river tile**. Rivers override the base biome's surface type with water but retain the underlying biome for resource/encounter purposes.

River width varies naturally with the noise function. Rivers in swamp biomes widen (threshold drops to `0.88`). Rivers in mountain biomes narrow (threshold rises to `0.95`).

## 5. Elevation

### 5.1 Elevation Computation

Elevation is a continuous value per tile, derived from the same elevation noise used for biome selection, but with additional biome-specific shaping applied:

```ts
tile.elevation = baseElevation(elevationNoise) + biomeElevationModifier(biome)
```

| Biome | Base Range | Modifier | Result Range |
|-------|-----------|----------|-------------|
| Plains | `[-0.5, 0.0]` | `+0.2` | `[0, 50]` mapped |
| Forest | `[-0.3, 0.3]` | `+0.3` | `[20, 60]` mapped |
| Mountain | `[0.5, 1.0]` | `+0.5` | `[70, 100]` mapped |
| Swamp | `[-1.0, -0.3]` | `+0.0` | `[0, 20]` mapped |
| Tundra | `[-0.2, 0.5]` | `+0.2` | `[10, 60]` mapped |
| Desert | `[-0.3, 0.3]` | `+0.1` | `[5, 45]` mapped |
| Taiga | `[0.0, 0.5]` | `+0.3` | `[25, 65]` mapped |

Final elevation is an integer in the range `[0, 100]`, where:
- `0` = sea level / lowest ground
- `100` = highest peak

### 5.2 Elevation and Gameplay

Elevation affects:
- **Movement cost**: Higher elevation differences between adjacent tiles increase movement cost
- **Visibility**: Higher elevation tiles can "see" further (relevant for fog of war)
- **Water flow**: Tiles at elevation 0 adjacent to rivers or swamp become water tiles
- **Visual rendering**: Elevation determines shadow direction and tile layering in the `.pixel` renderer

## 6. Surface Generation

### 6.1 Purpose

Surface generation determines the **visual surface type** and **traversability** of each tile based on its biome and elevation. This is analogous to Minecraft's surface rule system, simplified for 2D.

### 6.2 Surface Types

```ts
type SurfaceType =
  | "grass"       // Standard traversable ground
  | "dirt"        // Bare ground, paths
  | "sand"        // Desert, beaches
  | "stone"       // Exposed rock, mountain peaks
  | "snow"        // Frozen ground
  | "water"       // Lakes, rivers, ocean
  | "mud"         // Swamp ground
  | "ice"         // Frozen water
  | "pine_floor"  // Needle-covered forest floor
  ;
```

### 6.3 Surface Rules

Surface type is determined by biome, elevation, and a local noise function for variation:

| Biome | Elevation Range | Surface Type | Notes |
|-------|----------------|--------------|-------|
| Plains | 0–10 | `grass` | Low grass with wildflowers |
| Plains | 10–50 | `grass` | Standard grassland |
| Forest | 0–30 | `grass` | Forest floor |
| Forest | 30–60 | `dirt` | Dense canopy area |
| Mountain | 70–85 | `stone` | Rocky slopes |
| Mountain | 85–100 | `snow` | Snow-capped peaks |
| Swamp | 0–5 | `water` | Standing water |
| Swamp | 5–20 | `mud` | Soggy ground |
| Tundra | 0–20 | `ice` | Frozen lakes |
| Tundra | 20–60 | `snow` | Frozen ground |
| Desert | 0–45 | `sand` | Uniform sand |
| Taiga | 0–30 | `pine_floor` | Needle-covered ground |
| Taiga | 30–65 | `snow` | Higher altitude snow |

A **surface variation noise** (seed: `worldSeed + 4`) adds local randomness — e.g., patches of dirt within grass, or scattered stones on mountain slopes. When `surfaceVariationNoise > 0.7`, apply the biome's secondary surface type.

## 7. Resource Distribution

### 7.1 Purpose

Resources are items that can be gathered from tiles. Resource placement is biome-driven with noise-controlled density.

### 7.2 Resource Types

```ts
type ResourceType =
  | "wood"       // Trees (forest, taiga)
  | "stone"      // Rock deposits (mountain, plains)
  | "iron"       // Metal ore (mountain)
  | "herb"       // Medicinal plants (forest, swamp)
  | "fish"       // Water tiles
  | "clay"       // Riverbanks, swamp
  | "crystal"    // Rare, mountain peaks
  | "peat"       // Swamp fuel source
  | "fur"        // Tundra (from fauna, not tile — but spawn marker)
  ;
```

### 7.3 Resource Placement

Each biome defines a **resource table** with density thresholds. A per-resource noise function determines placement:

```ts
resourceNoise(x, y, resourceId) = octaveNoise(x, y, octaves: 2, persistence: 0.6, seed: worldSeed + 100 + hash(resourceId))
```

| Biome | Resource | Density Threshold | Notes |
|-------|----------|------------------|-------|
| Forest | `wood` | `> 0.1` (abundant) | Most tiles have trees |
| Forest | `herb` | `> 0.6` | Scattered clearings |
| Plains | `stone` | `> 0.7` | Occasional outcrops |
| Mountain | `stone` | `> 0.2` | Very common |
| Mountain | `iron` | `> 0.75` | Moderate deposits |
| Mountain | `crystal` | `> 0.95` | Rare, high elevation only (>90) |
| Swamp | `herb` | `> 0.3` | Rich in medicinal plants |
| Swamp | `clay` | `> 0.5` | Riverbanks and shallows |
| Swamp | `peat` | `> 0.6` | Fuel source |
| Tundra | `stone` | `> 0.5` | Exposed rock |
| Desert | `stone` | `> 0.8` | Rare sandstone |
| Taiga | `wood` | `> 0.2` | Pine forests |
| River tiles | `fish` | `> 0.3` | Along waterways |
| River tiles | `clay` | `> 0.6` | Riverbanks |

A tile's `resources` array contains all resources whose noise exceeds the threshold at that position.

## 8. Structure Generation

### 8.1 Overview

Structures are pre-defined or procedurally assembled multi-tile features placed during generation. They override the underlying terrain and resources. This adapts Minecraft's spacing/separation system for 2D.

### 8.2 Structure Placement Algorithm

Each structure type has placement parameters:

```ts
type StructurePlacement = {
  id: StructureID;
  spacing: number;       // Average distance between placements (in chunks)
  separation: number;    // Minimum distance between placements (in chunks), must be < spacing
  frequency: number;     // Probability [0.0, 1.0] that a valid position is used
  biomes: BiomeType[];   // Allowed biomes
  minElevation?: number; // Optional elevation constraint
  maxElevation?: number;
};
```

**Placement grid**: The world is divided into a grid of `spacing × spacing` chunk cells. Within each cell, a candidate position is determined by hashing the cell coordinates with the world seed. If the candidate is at least `separation` chunks from all neighbors' candidates, and the biome/elevation match, and a frequency check passes, the structure is placed.

```
candidateChunk(cellX, cellY) = (
  cellX * spacing + hash(worldSeed, cellX, cellY) % (spacing - separation),
  cellY * spacing + hash(worldSeed, cellX, cellY, 1) % (spacing - separation)
)
```

### 8.3 Structure Definitions (v1)

| Structure | Spacing | Separation | Frequency | Biomes | Description |
|-----------|---------|-----------|-----------|--------|-------------|
| `village` | 12 | 4 | 0.6 | plains, forest | Small NPC settlement (3×3 to 5×5 tiles) |
| `ruins` | 8 | 3 | 0.5 | any | Crumbling stone structure (2×2 to 4×4) |
| `cave_entrance` | 6 | 2 | 0.7 | mountain, forest | Entrance to underground area (1×1 marker) |
| `shrine` | 16 | 6 | 0.4 | any | Magical waypoint (1×1 to 2×2) |
| `watchtower` | 10 | 4 | 0.5 | plains, mountain, tundra | Tall structure, visibility bonus (2×2) |
| `bridge` | — | — | — | river tiles | Placed at river crossings near paths (special placement, see §8.4) |

### 8.4 Special Placement: Bridges and Paths

Bridges don't use the spacing/separation grid. Instead, they are placed where a **path** (generated as a low-cost traversal line between two structures) crosses a river. Path generation uses a simplified A* between nearby structures after structure placement is complete.

### 8.5 Structure Templates

Each structure has a **template** — a small tile grid that overrides the procedural terrain:

```ts
type StructureTemplate = {
  id: StructureID;
  width: number;           // tiles
  height: number;          // tiles
  tiles: StructureTile[];  // override data per tile
  npcs?: NPCSpawn[];       // NPCs to place
  loot?: LootTable[];      // items available in the structure
};

type StructureTile = {
  localX: number;
  localY: number;
  surface: SurfaceType;
  features: FeatureType[];
  walkable: boolean;
};
```

### 8.6 Scenario Override

Scenarios (per `loreweave_spec.md` §4.6) can **reserve** specific chunk regions for authored content. During the structure generation stage, reserved regions skip procedural structure placement entirely. The scenario's structures take precedence.

## 9. Feature Placement

### 9.1 Overview

Features are single-tile or small decorative elements placed after structures. They add visual variety and minor gameplay interactions without the multi-tile complexity of structures.

### 9.2 Decoration Steps

Features are placed in **5 ordered steps**, from large/important to small/decorative. Earlier steps take priority — a tile that already has a feature from an earlier step won't receive one from a later step.

| Step | Name | What Gets Placed |
|------|------|-----------------|
| 1 | **Terrain features** | Boulders, cliffs, ponds, sand dunes |
| 2 | **Vegetation (major)** | Trees, large bushes, cacti, dead trees |
| 3 | **Vegetation (minor)** | Flowers, mushrooms, tall grass, reeds, moss |
| 4 | **Ground detail** | Fallen logs, leaf piles, pebbles, bones |
| 5 | **Ambient markers** | Fog patches (swamp), snow drifts (tundra), heat shimmer (desert) |

### 9.3 Feature Types

```ts
type FeatureType =
  // Terrain features
  | "boulder" | "cliff" | "pond" | "sand_dune"
  // Major vegetation
  | "oak_tree" | "pine_tree" | "dead_tree" | "bush" | "cactus" | "willow_tree"
  // Minor vegetation
  | "flowers" | "mushrooms" | "tall_grass" | "reeds" | "moss" | "fern"
  // Ground detail
  | "fallen_log" | "leaf_pile" | "pebbles" | "bones" | "campfire_remains"
  // Ambient
  | "fog" | "snow_drift" | "heat_shimmer"
  ;
```

### 9.4 Biome Feature Tables

Each biome defines which features can appear and at what density:

**Forest**:
- `oak_tree` (40% of tiles), `bush` (15%), `flowers` (10%), `mushrooms` (5%), `tall_grass` (20%), `fallen_log` (3%), `leaf_pile` (5%), `boulder` (2%)

**Plains**:
- `tall_grass` (40%), `flowers` (15%), `bush` (5%), `boulder` (3%), `pebbles` (5%)

**Mountain**:
- `boulder` (25%), `cliff` (10%), `pine_tree` (8%), `pebbles` (15%), `snow_drift` (10% if elevation > 85)

**Swamp**:
- `willow_tree` (20%), `reeds` (25%), `mushrooms` (15%), `moss` (20%), `fog` (10%), `pond` (8%), `bones` (2%)

**Tundra**:
- `snow_drift` (30%), `dead_tree` (5%), `boulder` (10%), `pebbles` (10%), `bones` (3%)

**Desert**:
- `cactus` (8%), `sand_dune` (20%), `pebbles` (10%), `bones` (5%), `heat_shimmer` (8%)

**Taiga**:
- `pine_tree` (35%), `bush` (10%), `fern` (15%), `mushrooms` (8%), `fallen_log` (5%), `moss` (10%), `snow_drift` (5%)

### 9.5 Feature Placement Algorithm

For each tile in the chunk, iterate through decoration steps in order:

```
for each decorationStep in [1..5]:
  for each tile in chunk:
    if tile already has feature from earlier step: skip
    if tile is structure tile: skip
    candidates = biomeFeatureTable[tile.biome][decorationStep]
    for each candidate in candidates:
      roll = positionalRNG(worldSeed, tile.x, tile.y, candidate.id)
      if roll < candidate.density:
        tile.features.push(candidate.type)
        break  // one feature per step per tile
```

## 10. Spawn Generation

### 10.1 Purpose

The spawn stage determines where NPCs and creatures may appear. It does not place actual entities — it marks **spawn points** that the runtime game loop uses to populate the world as players approach.

### 10.2 Spawn Rules

```ts
type SpawnRule = {
  entityType: string;      // e.g., "wolf", "goblin", "deer"
  biomes: BiomeType[];
  density: number;         // average spawns per chunk
  minGroupSize: number;
  maxGroupSize: number;
  minElevation?: number;
  maxElevation?: number;
  requiresFeature?: FeatureType[]; // must be near specific features
  avoidStructures?: boolean;       // don't spawn near structures
};
```

### 10.3 Spawn Table (v1)

| Entity | Biomes | Per Chunk | Group Size | Notes |
|--------|--------|-----------|------------|-------|
| `deer` | forest, plains, taiga | 1.5 | 2–4 | Passive, flees from players |
| `wolf` | forest, taiga, tundra | 0.8 | 2–3 | Neutral, hunts deer |
| `goblin` | forest, swamp, mountain | 0.5 | 3–6 | Hostile, near ruins/caves |
| `bandit` | plains, forest | 0.3 | 2–4 | Hostile, near paths |
| `bear` | forest, mountain, taiga | 0.3 | 1 | Neutral, territorial |
| `snake` | swamp, desert | 0.6 | 1–2 | Hostile, hidden |
| `merchant` | plains | 0.1 | 1 | Passive, travels between villages |

### 10.4 Spawn Point Placement

For each entity in the spawn table whose biome matches:

1. Determine count: `positionalRNG(worldSeed, cx, cy, entityType) × density × 2` (Poisson-like distribution)
2. For each spawn, pick a random tile within the chunk using positional RNG
3. Validate: correct biome, not inside a structure (if `avoidStructures`), required features nearby
4. Record the spawn point in the chunk data

Actual entity instantiation happens at runtime when players enter the chunk's activation radius.

## 11. Generation Pipeline Summary

The complete pipeline for generating a single chunk:

```
Input: worldSeed, chunkCoord (cx, cy)

Step 1 — BIOMES
  For each tile (x, y) in chunk:
    Compute temperature(x, y), moisture(x, y), elevationNoise(x, y)
    Select biome via closest-match in 3D parameter space
    Apply biome blending (4-tile radius)
    Apply river overlay

Step 2 — ELEVATION
  For each tile:
    Compute elevation from elevationNoise + biome modifier
    Map to integer [0, 100]

Step 3 — SURFACE
  For each tile:
    Determine surface type from biome + elevation + variation noise
    Set traversability

Step 4 — RESOURCES
  For each tile:
    Evaluate resource noise per resource type
    Populate resources array based on biome resource table

Step 5 — STRUCTURES
  Check structure placement grid for this chunk
  If candidate position exists and passes all checks:
    Apply structure template (overrides tiles)
  Check for scenario reservations (override procedural structures)
  Store structure references from neighboring chunks

Step 6 — FEATURES
  For each decoration step (1–5):
    For each tile not occupied by structures or earlier features:
      Roll against biome feature table
      Place feature if roll succeeds

Step 7 — SPAWNS
  For each entity in spawn table matching chunk biomes:
    Determine spawn count and positions
    Validate placement constraints
    Record spawn points

Output: Chunk with fully populated WorldTile[] array
```

## 12. Determinism Guarantees

Unlike Minecraft, where chunk generation order can cause minor non-determinism (overlapping feature placement), LoreWeave enforces strict determinism:

1. **No cross-chunk feature bleeding**: Features are placed within chunk boundaries only. The 4-tile biome blend reads neighbor *biome data* (stage 1) but does not write to neighbor tiles.
2. **Positional RNG everywhere**: Every random decision uses `positionalRNG(seed, x, y, context)`, never a shared mutable RNG state.
3. **Idempotent stages**: Running a stage twice on the same input produces the same output.
4. **Structure reservation before feature placement**: Structures claim tiles before features run, so there's no race condition.

**Invariant**: For any given `(worldSeed, cx, cy)`, the output `Chunk` is byte-identical regardless of what other chunks have been generated, in what order, or how many times.

## 13. Performance Considerations

### 13.1 Noise Caching

Noise functions are expensive. Cache noise values at chunk granularity — compute all 32×32 values for a noise layer in one pass rather than per-tile.

### 13.2 Lazy Generation

Chunks should only advance through stages as needed:
- **Far from players**: Generate through `biomes` only (enough for minimap)
- **Within load radius**: Generate through `features` (visual rendering)
- **Player enters chunk**: Generate through `spawns` (gameplay ready)

### 13.3 Chunk Storage

Once a chunk reaches `complete`, persist it. Never re-generate a completed chunk — player modifications (placed structures, gathered resources, killed spawns) would be lost.

## 14. Extensibility

### 14.1 Adding Biomes

To add a new biome:
1. Define its `(temperature, moisture, elevationNoise)` center and ranges in the biome selection table (§4.2)
2. Add surface rules (§6.3)
3. Add resource table entries (§7.3)
4. Add feature table entries (§9.4)
5. Add spawn table entries (§10.3)

No code changes needed beyond data — the generation pipeline is data-driven.

### 14.2 Adding Structures

To add a new structure:
1. Define a `StructurePlacement` entry (§8.2)
2. Create a `StructureTemplate` (§8.5)
3. The placement algorithm handles the rest

### 14.3 Adding Features

To add a new feature:
1. Add to the `FeatureType` union
2. Add entries to the relevant biome feature tables (§9.4)
3. Provide `.pixel` art assets for the renderer

## 16. Open Questions

These items need decisions before or during implementation:

1. **Chunk size confirmation**: 32×32 is specified here. The core spec says "16×16 or 32×32". Need to commit to one. Recommendation: 32×32 for richer generation at slight memory cost.
2. **Water body generation**: Rivers are covered (§4.4). Lakes and oceans are not yet specified. Should oceans exist, or is LoreWeave landlocked? If oceans exist, add a `continentalness` noise parameter.
3. **Underground/dungeon generation**: `cave_entrance` structures are defined as markers. The actual underground generation (dungeon layouts, underground biomes) is deferred — it's a significant subsystem.
4. **Elevation cost formula**: §5.2 states elevation affects movement cost but doesn't define the formula. The rules engine owns this, but the spec should define the data contract.
5. **Noise library choice**: Perlin vs. Simplex vs. OpenSimplex2. Simplex is generally preferred for 2D (no axis-aligned artifacts). Recommend OpenSimplex2 for licensing clarity.
